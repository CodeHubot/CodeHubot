## 🎯 快速开始

### 新成员必读
如果你是新加入的开发者，建议按以下顺序阅读：

1. 📖 阅读本文档了解全局
2. 🔴 学习**核心规范**（文档 1-4）
3. 🟡 了解**流程规范**（文档 5-7）
4. 🟢 参考**架构规范**（文档 8-10）

### 开发前检查清单

在开始新功能开发前，请确保：

- [ ] 已阅读相关的开发规范文档
- [ ] 了解项目的整体架构和目录结构
- [ ] 熟悉 API 调用规范和响应格式
- [ ] 掌握代码风格要求
- [ ] 了解数据库设计规范
- [ ] 明确 Git 提交信息格式

### 代码提交前检查清单

在提交代码前，请确保：

- [ ] 代码符合风格规范
- [ ] 已添加必要的注释和文档字符串
- [ ] 后端 API 使用统一响应格式 (`success_response`)
- [ ] 数据库变更已生成 SQL 脚本（放在 `SQL/update/` 目录）
- [ ] 已测试核心功能
- [ ] 提交信息符合规范格式
- [ ] 没有遗留 console.log 或调试代码
- [ ] 没有硬编码的敏感信息

## 📋 规范优先级说明

- 🔴 **高优先级**: 必须严格遵守，影响系统稳定性和代码质量
- 🟡 **中优先级**: 建议遵守，提升开发效率和协作体验
- 🟢 **低优先级**: 参考建议，优化代码质量和性能

## 🔄 规范更新

开发规范会随着项目发展持续更新，请定期查看。

- **最后更新**: 2025-12-18
- **维护者**: 项目组
- **反馈渠道**: 通过 Issue 或 Pull Request 提出改进建议

## 💡 规范制定原则

1. **实用性**: 规范应该有助于解决实际问题，而不是增加负担
2. **一致性**: 团队成员遵循统一的规范，降低沟通成本
3. **灵活性**: 规范不是教条，特殊情况可以灵活处理，但需要说明理由
4. **渐进性**: 规范逐步完善，不要一开始就过于严格

## 🤝 参与改进

如果你发现规范中的问题或有改进建议：

1. 在团队会议中讨论
2. 提交 Issue 说明问题
3. 提交 Pull Request 改进文档
4. 与项目维护者沟通

---

**记住**: 规范是为了让开发更顺畅，而不是制造障碍。如果某个规范让你感觉不合理，请及时反馈！
